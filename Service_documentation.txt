1. INTRODUCTION
───────────────

1.1 Project Overview
────────────────────
Ninja Deliveries — Service Management System is a full-stack web application designed to streamline and digitize operations for service-based businesses such as electricians, plumbers, cleaners, and technicians.
 It replaces manual processes with a centralized platform for managing bookings, workforce, and customer interactions.
 
The system enables real-time booking, efficient technician assignment, payment tracking, and instant notifications, improving operational efficiency and service quality.
 It is developed as a modular sub-system within the Ninja Deliveries ecosystem, with its own authentication, database structure, and dedicated dashboard, ensuring scalability, data isolation, and ease of maintenance.

1.2 Purpose and Objectives
──────────────────────────
The primary purpose of the system is to provide a centralized and efficient platform for managing service operations digitally. It aims to improve operational transparency, reduce manual effort, and enhance overall service delivery.

Objective                    | Description
-----------------------------|------------------------------------------------
Digital Booking Management   | Replace manual booking methods with a structured, real-time booking system
Worker Assignment            | Assign technicians based on skills, availability, and service requirements
Live Notifications           | Provide instant alerts for new bookings and status updates
Package Subscriptions        | Support flexible service plans such as daily, weekly, and monthly packages
Admin Oversight              | Enable centralized monitoring and control for administrators
Payment Tracking             | Record, manage, and generate reports for all service-related transactions
Feedback Collection          | Collect and analyze customer ratings and reviews for service improvement

1.3 Key Features
────────────────
Real-time Booking Alerts
Instant notifications for new bookings using audio signals and browser push notifications

Secure booking completion through One-Time Password (OTP) validation

3. SYSTEM ARCHITECTURE

──────────────────────

3.1 High-Level Architecture

────────────────────────────

The system follows a client-server architecture using Firebase as a serverless backend platform.
The frontend is developed with React, leveraging the Context API for global state management and React Router for seamless client-side navigation.

This architecture enables real-time data synchronization, reduces backend maintenance overhead, and supports scalable application growth.

3.2 Master-Slave Architecture

──────────────────────────────

The system implements a Master-Slave architecture for managing service categories and services.
This approach ensures centralized control while allowing company-level customization.

Master Layer (Admin Controlled):

service_categories_master
service_services_master

Slave Layer (Company-Specific):

service_categories (linked via masterCategoryId)
service_services (linked via masterServiceId)

Key Benefits:

Centralized updates propagate automatically across all companies
Consistent synchronization of images and service status
Cascading logic maintains data integrity across related entities
3.3 Real-Time Data Flow

──────────────────────────

The system operates on an event-driven, real-time data flow model powered by Firebase.

Flow Sequence:
Customer creates booking → Firestore listener (onSnapshot) triggers → Notification system processes audio, push, and toast alerts → Dashboard updates instantly

This ensures that all users receive immediate updates, enabling faster response times and synchronized system behavior across all active sessions.

4. DATABASE DESIGN
──────────────────

Core Collections:

• service_company          → Company profile (Document ID = Auth UID)
• service_bookings         → All customer bookings with full lifecycle
• service_workers          → Technician profiles and assignments
• service_categories       → Company-specific categories
• service_services         → Company-specific services
• service_categories_master→ Global category templates
• service_services_master  → Global service templates
• service_payments         → Payment records
• service_feedback         → Customer reviews and ratings
• service_availability     → Offline windows scheduling
• service_activity_logs    → Audit trail for admin actions

Collection Relationships:
service_company (1) → service_bookings (N), service_workers (N), service_categories (N), etc.

5. MODULE-WISE DOCUMENTATION (Refined & Professional)

────────────────────────────

This section outlines the core modules of the system, each designed to handle specific functional responsibilities within the service management workflow.

5.1 Service Dashboard Shell

Provides the primary layout structure of the application, including sidebar navigation, company branding (logo), real-time booking indicators, and a fully responsive interface for different screen sizes.

5.2 Overview Module

Displays key operational insights such as today’s booking summary, weekly revenue visualization, and quick action shortcuts for common administrative tasks.

5.3 Bookings Module

Manages the complete booking lifecycle, including creation, assignment, status updates, OTP-based completion verification, auto-expiry handling, and rejection workflows. Supports both single and package-based bookings.

5.4 Slots / Calendar Module

Offers a visual calendar interface to manage booking schedules, track daily booking volumes, define offline availability windows, and toggle service availability.

5.5 Technicians Module

Handles technician (worker) management, including assignment, availability, and status control. Implements cascading deactivation logic to ensure service consistency when a worker is disabled.

5.6 Services & Categories Modules

Enables management of service categories and individual services with support for master data synchronization and flexible pricing models, including package configurations.

5.7 Payments Module

Tracks all service-related transactions, provides revenue insights, and supports export of financial data in CSV format for reporting and analysis.

5.8 Feedback Module

Collects customer ratings and reviews, and calculates average feedback metrics to monitor service quality and performance.

5.9 Service Admin Panel

Provides a centralized dashboard for super administrators to manage companies, monitor system-wide activities, and access detailed activity logs.

5.10 Restaurant Module

A parallel subsystem that allows restaurant owners to manage menu categories and related data within the same platform architecture.

7. KEY WORKFLOWS / USE CASES (Refined & Professional)

────────────────────────────

This section outlines the core operational workflows that define how the system functions in real-time scenarios.

7.1 End-to-End Booking Flow

Describes the complete lifecycle of a service booking from initiation to completion.

Flow:
Customer places booking → System generates real-time notification → Admin assigns technician → Technician performs service → OTP verification → Booking marked as completed

7.2 Package Booking Flow

Handles recurring service bookings grouped under a single package.

Flow:
Customer selects package → System creates grouped bookings → Bookings are processed on scheduled dates → Status managed individually for each entry

7.3 Worker Cascading Deactivation Logic

Ensures system consistency when a technician is deactivated.

Logic:
If a worker is deactivated → All associated services linked to that worker are automatically disabled → Prevents invalid or unserviceable bookings

7.4 Real-Time Notification Pipeline

Defines how the system delivers instant updates to users.

Flow:
Booking event triggered → Firestore listener activates → Notification system processes event → Audio alert + push notification + UI update
